你这个质疑**非常好，而且你已经摸到 WireGuard 的“真实工作机制”了**。
我们现在就把这件事**彻底讲清楚**，一次性把「外层 UDP 包」和「内层 IP 包」的关系说死，否则这一步如果糊涂，后面所有 SDP / NAT 推理都会错。

我先给结论，然后用你给的例子**逐层拆包**。

---

## 一、先给结论（非常关键）

> **你画的这个包形态：**
>
> ```
> [ UDP
>   [ sip = 10.0.0.2 , dip = 10.0.0.3 ]
>   [ 加密数据
>     [ sip = 10.248.10.1 , dip = 10.213.148.96 ]
>   ]
> ]
> ```
>
> ❌ **这是不对的**

### ✅ 正确的结论是

> **WireGuard 隧道内的“内层 IP 包”，其源地址一定是 WG 接口的 IP（比如 10.0.0.2），而不可能是客户端物理网卡 IP（10.248.10.1）。**

**WireGuard 不会把“物理网卡 IP”封装进隧道里。**

---

## 二、为什么你会产生这个疑问（而且很合理）

你的直觉其实来自 **IPsec / GRE / VXLAN 的某些场景**，但 **WireGuard 的模型不一样**。

你在想的是：

> “我机器真实 IP 是 10.248.10.1，
> 那我访问 10.213.148.96，
> 是不是把这个真实包‘原样’塞进隧道？”

**WireGuard 明确不是这么做的。**

---

## 三、WireGuard 的核心原则（一句话定死）

> **WireGuard 是一个 L3 VPN（三层隧道），不是 L2，也不是“透明封装”**

它的基本规则是：

> **进入 wg 接口的包，源 IP 就是 wg 接口的 IP**

这一步在**客户端本机**已经完成了。

---

## 四、用你的例子，把 Client 发包过程完整拆解一遍

### 假设条件（严格对齐你说的）

* Client 物理网卡 IP：`10.248.10.1`
* Client wg 接口 IP：`10.0.0.2`
* PE wg 接口 IP：`10.0.0.3`
* 目标服务：`10.213.148.96:8080`

---

### 🧱 第 1 步：应用层发起连接

```text
curl http://10.213.148.96:8080
```

应用层根本不知道 WireGuard。

---

### 🧭 第 2 步：路由决策（关键点）

Client 上一定存在类似路由：

```text
10.213.148.96/32 → wg0
```

于是内核决定：

> **这个包要走 wg0**

---

### 🔁 第 3 步：包进入 wg0（关键转折点）

此时内核做了一件**极其重要但你容易忽略的事**：

> **把包的源地址，改成 wg0 接口的地址**

也就是说，**内层包已经是：**

```text
SRC = 10.0.0.2
DST = 10.213.148.96
```

📌 **这一步不是 NAT，而是“从哪个接口出，就用哪个接口的 IP”**

和你从 `eth0` 出包，源地址是 `eth0 IP` 是一模一样的逻辑。

---

### 🔐 第 4 步：WireGuard 封装（加密）

现在 WireGuard 才开始工作。

它做的是：

```text
把这个 IP 包整体加密
```

然后外面再包一层 UDP：

```text
OUTER UDP:
  SRC = Client 物理 IP (10.248.10.1)
  DST = PE 物理 IP

INNER (加密):
  SRC = 10.0.0.2
  DST = 10.213.148.96
```

👉 **注意：**

* 物理 IP 只存在于外层 UDP
* 内层永远是 wg IP

---

## 五、PE 上解密后的真实包长什么样？

当 PE 收到 UDP 包并解密后，**内核看到的是：**

```text
SRC = 10.0.0.2
DST = 10.213.148.96
IN = wg0
```

📌 **绝不可能出现 `SRC=10.248.10.1`**

---

## 六、所以我们回头修正你最关心的 NAT 推理

现在你就能理解了👇

### 在 PE 的 POSTROUTING 里

```text
SRC = 10.0.0.2
DST = 10.213.148.96
```

所以：

```bash
iptables -t nat -A POSTROUTING \
  -s 10.0.0.0/24 \
  -d 10.213.148.0/24 \
  -j MASQUERADE
```

**逻辑是完全正确的**。

如果源地址是物理 IP，那这条规则反而根本匹配不到。
